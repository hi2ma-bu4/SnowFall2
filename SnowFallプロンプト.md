# SnowFall 言語：実行エンジン構築 & 逐次テスト駆動開発指令

あなたは、独自言語「SnowFall」の開発チームにおける**ランタイム・エンジニア**です。現在完了している「ASCII中間コード出力」を受け取り、仕様書に従って正しく解釈・実行するエンジンを構築します。

**※タスク E2（プロトタイプベースの動的実行）は本フェーズでは対象外とします。**

## 1. 開発プロトコル：1機能・1テストの徹底
本フェーズも**テスト駆動開発（TDD）**を厳守します。
1. **テストの追加/更新:** `npm run test` で失敗する、実行エンジンのテストケースを記述する。
2. **機能実装:** そのテストをパスするための最小限の実行ロジックを実装する。
3. **検証:** `npm run test` で成功を確認し、次の機能へ進む。

## 2. 実装タスクと仕様上の重要ポイント

詳細な仕様については、必ず **『SnowFall_Specification.md』** を常に参照しながら進めてください。

### タスク E1: ASCII中間コードのデシリアライザ（仕様 1.1 参照）
ASCII形式で圧縮された中間コードを読み込み、Rust側で実行可能な命令列（Instruction Set）へ高速に復元するロジックを実装してください。
* **仕様要件:** * プレーンテキスト形式の中間コードから命令・型・値を抽出すること。
    * デバッグオプションが有効な場合、命令に紐付いたソース位置情報を保持すること。

### タスク E3: 暗黙の型変換と比較演算の実装（仕様 3.2.1, 3.2.3 参照）
実行時における数値演算と論理比較のロジックを実装してください。
* **仕様要件:** * `==` 演算子における `String > Float > Int > Char > Boolean` の優先順位に基づく型変換。
    * `+` 演算子における、オペランドの一方が `String` の場合の強制文字列連結（`__to_string()` の呼び出しを含む）。
    * `and` / `or` 演算子における、両辺の強制評価と `Boolean` 返却の保証。

### タスク E4: Rust-TS 双方向連携プロトコルの実体（仕様 3.6 参照）
RustからTS側の関数を呼び出す `invokeHostFunction` メカニズムを実装してください。
* **仕様要件:** * `HostRequest` オブジェクト（operation, args, requires_return）の生成とポインタによるTSへの受け渡し。
    * TSからの `HostResponse` を受け取り、成功時は結果を `SnowValue` に変換、失敗時は `SnowFallError` を構築して実行を中断すること。

### タスク E5: 再帰上限チェック（仕様 3.5 参照）
* **仕様要件:** * 関数（`function`）またはサブルーチン（`sub`）の呼び出しごとにスタック深さをカウントし、上限（初期値 10,000）を超えたら即座にエラー（SFコード）を出すこと。



## 3. テスト駆動開発（TDD）の更新
各機能追加ごとに、以下のテストを `npm run test` でパスさせてください。
1. **連結テスト:** `Int(10) + String(" units")` が `"10 units"` になること。
2. **型変換比較テスト:** `Int(1) == String("1")` が `true` になること。
3. **ホスト連携テスト:** `print("test")` 実行時に `HostRequest` が正しく発行され、TS側で受信できること。
4. **再帰エラーテスト:** 無限再帰させた際、Rust側で検知して `SnowFallError` が返却されること。

実装の詳細は常に **『SnowFall_Specification.md』** を正典として参照し、1機能ごとにテスト結果を報告してください。
