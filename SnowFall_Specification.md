# SnowFall 言語 仕様書

SnowFallは、TypeScriptの厳密な型付けとPythonの柔軟性を融合させた、コンパイラ方式の独自言語です。


## 0. 注意点
`npm run build`でプロジェクト全体を検証&コンパイルします。

`npm run build`以外の方法でコンパイルしないでください。


## 1. システム・開発環境仕様

### 1.1. 開発構成

| 項目                           | 詳細                                                                         | 補足                                                                               |
| :----------------------------- | :--------------------------------------------------------------------------- | :--------------------------------------------------------------------------------- |
| **開発言語**                   | Rust (WebAssembly) および TypeScript                                         | 根幹・解析はRust (Wasm) 側で実装し、JS/TSの実行を減らすことで速度を確保します。    |
| **実行環境**                   | Webブラウザ(クライアントサイド完結)                                          | すべての実行(検証、コンパイル、実行)はWeb上で動作します。                          |
| **動作段階**                   | 検証 (Static Analysis)、コンパイル (Compilation)、実行 (Execution) の3段階。 | 各段階は独立した処理として定義され、中間コードを介してデータの受け渡しを行います。 |
| **コンパイル結果(中間コード)** | 独自の**中間コード**(プレーンテキスト形式)(ASCII範囲)                        | ASCII範囲であれば圧縮を許容します。中間コードの具体的な仕様は別途定義。            |

### 1.2. モジュール設計と独立性

* **機能独立性:** 検証&コンパイル機能と、実行機能は**それぞれ独立して動作**するようにファイルを分離します。
* **共通利用:** `util`などの共通利用モジュールは、TreeShakingを考慮して許容します。
* **言語要素の分離:** 言語機能(`Int`, `String`などの型、`if`, `for`などの構文)はそれぞれ個別にファイルに分割し、依存度を下げ、**拡張を容易**にします。
* **補足:** 各機能(`if`, `for`, `while` など)のパーサおよびコード生成ロジックは、個別のファイルに分離します。
* **補足2:** 型などの既存クラスは個別のファイルで管理し、管理コストを下げます。
* **クラス構造:** **TypeScript側**のラッパー(インターフェース)として、検証、コンパイル、実行の各機能は**クラス**で作成し、**コンストラクタ**で初期設定(オプション)を設定可能とします。

### 1.3. エラー処理とデバッグ補助

* **エラー情報:** コンパイルエラーおよび実行時エラーでは、**何行目の何文字目**で問題が発生したかを示す情報を表示可能とします。
* **エラークラス:** エラーは独自クラス(`SnowFallError`)で出力させます。
    * `SnowFallError`オブジェクトは、以下の必須プロパティを持ちます。
        | プロパティ名 |      型       | 必須/任意 | 説明                                                           |
        | :----------- | :-----------: | :-------: | :------------------------------------------------------------- |
        | `type`       |    String     |   必須    | エラーの種類 (例: CompilationError, RuntimeError, SyntaxError) |
        | `message`    |    String     |   必須    | ユーザー向けの簡潔な説明                                       |
        | `code`       |    String     |   必須    | エラーを一意に識別するためのコード (例: SF001, SF205)          |
        | `line`       |    Number     |   必須    | エラーが発生した行番号(1ベース)                                |
        | `column`     |    Number     |   必須    | エラーが発生した文字数(1ベース)                                |
        | `trace`      | Array<String> |   必須    | スタックトレース(関数呼び出しの履歴。Rust側から取得)           |
        | `context`    |    Object     |   任意    | エラーに関連する追加情報 (例: 期待された型、見つかった型)      |
    * 出力側TSのエラークラスはtypeを元に`SnowFallError`を拡張して使用し、エラー定義は1ファイルにまとめます。
* **エラー定義と拡張:**
    * **エラー定義の統合:** すべてのエラーコード(例：SF001)とそれに対応するデフォルトメッセージは、Rust/TS間で共通利用される単一のJSONファイルとして管理します。
    * **拡張:** 他のエラークラスは、SnowFallErrorを継承(TS側でラッパークラスとして定義)するか、typeプロパティを変えることで識別可能とします。
* **デバッグオプション:** コンパイル結果のデータサイズが増える場合は、`SnowFall`実行コンストラクタオプションで**デバッグ補助情報**の出力を変更できるようにします。
* **エラー表示の補助:** デバッグオプション(1.3. 節)が有効な場合、以下の補助情報を中間コードに追加します。
    * **デバッグ補助情報:** **ソースコードのASTノードと対応する行番号/文字数のマッピング**を、中間コードのメタデータに含めます。これにより、実行時にエラーが発生した場合、中間コードのどの命令がソースコードのどの位置に対応するかを正確に特定できます。

### 1.4. 開発規約

* **ドキュメント:** JSDocやRustのドキュメンテーションコメントは、**すべての関数、機能**について**日本語**で記載します。

## 2. 🚀 動作フロー(Rust/TS連携)

SnowFallの実行は、以下の手順でクライアントサイドで完結します。

| ステップ | 担当     | 処理内容                                                                                                           |
| :------- | :------- | :----------------------------------------------------------------------------------------------------------------- |
| **1**    | ユーザー | SnowFall言語のソースを作成します。                                                                                 |
| **2**    | TS       | ソースコードの**文字列**をRust (Wasm) に委託します。                                                               |
| **3**    | Rust     | **静的検証**と**コンパイル**を実施し、**検証結果(エラー/警告リスト)**と**中間コード**をTSに返却します。            |
| **4**    | TS       | 返却された中間コードを取得します(エラーがあればユーザーに結果を提供)。                                             |
| **5**    | TS       | 中間コードをRust (Wasm) に委託します。                                                                             |
| **6**    | Rust     | 中間コードを解析し、実行します。必要に応じてTSと連携します(外部ライブラリなどの呼び出し、ユーザー拡張機能の実行)。 |
| **7**    | TS       | Rust実行結果をユーザーに返却します(必要に応じてRustとの対話を担います)。                                           |
| **8**    | ユーザー | 最終結果を受け取ります。                                                                                           |

## 3. 📝 言語仕様

### 3.1. 基本構文と型

* **基本構文:** Cライクの `{}` を使用した言語です。
    * `{}` はスコープの境目とします。
* **変数宣言:** `(型名) (変数名) = (値);` の形式で記載します。
    * 例: `Int foo = 0;`
* **型の種類:**
    * **プリミティブ型:** `Int`, `Long`, `Float`, `Char`, `String`, `Boolean` など。
    * **オブジェクト型:** `Function`, `Sub`, `Class`, `Array`, `Tuple`, `Dictionary`, `Object` など。
* **プロトタイプベース:** すべての型はJavaScriptと同様の構造(`__proto__`)を持つ**プロトタイプベース**で動作します。
    * すべての機能(型、クラス、関数)はSnowFallオブジェクトで作成されており、`String.__proto__.foo = newFunc;` のように**プロトタイプから機能を直接編集可能**です。
    * **内部オブジェクトモデル(Rust側):**
        * SnowFallのすべてのインスタンス、型、関数は、Rust側で定義される共通の構造体 SnowObject または SnowValue のインスタンスとして表現されます。これにより、すべてのものが統一的に扱えます。
        | プロパティ |              型               | 説明                                                                                                   |
        | :--------: | :---------------------------: | :----------------------------------------------------------------------------------------------------- |
        |  type_id   |              U32              | "オブジェクトの具体的な型(Int, String, Arrayなど)を一意に識別するID。"                                 |
        |    data    |          Enum/Union           | オブジェクトが持つ実際の値(例: `Int`型の数値、`String`型のUTF-8バイト配列)。                           |
        | __proto__  |      Option<SnowObject>       | **継承元のプロトタイプオブジェクトへの参照**。nullでない限り、親のメソッドやプロパティへの参照を持つ。 |
        | properties | "HashMap<String, SnowObject>" | インスタンス固有のプロパティ(フィールド、メソッド)。                                                   |
    * **プロトタイプチェーンと編集の機構:** SnowFallは、JavaScriptのプロトタイプチェーンの動作を模倣します。
        1. **プロパティ/メソッドの検索:** インスタンスのプロパティにアクセスする際、まず自身の`properties`を検索します。見つからなければ、`__proto__`参照をたどって親の`properties`を順次検索します。
        2. **プロトタイプ編集の対象:** 編集が可能なのは、`Object`、`Int`、`String`などの**型そのものが持つプロトタイプオブジェクト**です。
            * 例: `String.__proto__.foo = newFunc;`
        3. **実行時編集の実現:**
            * `String.__proto__`などのプロトタイプオブジェクトは、Rust側で共有され、変更可能なメモリ領域に格納されます。
            * `String.__proto__.foo = value;`のような代入が行われると、Rustはメモリ上の共有されたプロトタイプオブジェクトの `properties`マップを動的に更新します。
            * この更新は、**その型から作成されたすべてのインスタンス**に、即座に影響を与えます。
    * **動的・静的アクセス権の分離:** 型付けの厳密さを維持するため、動的なプロトタイプ編集と静的な検証を以下のように区別します。
        | アクセス種別 | 挙動                                                                                                                                            |
        | :----------- | :---------------------------------------------------------------------------------------------------------------------------------------------- |
        | 静的検証     | コンパイル時、ソースコードに記述されたメソッド呼び出し (`myString.toUpperCase()`)は、現在の String 型の定義に基づいて厳密に型チェックされます。 |
        | 実行時       | `String.__proto__.foo = value;`のように実行時に追加されたメソッドは、コンパイル時には検証されません。実行時に検索ロジックを通じて解決されます。 |
        * **設計上の決定:** コンパイル時に存在しないプロトタイプメソッドをソースコードで呼び出した場合(例：`myString.newMethod()`を定義前に書いた場合)、コンパイルエラーとします。動的編集は、主にライブラリやデバッグ用途に限定されます。
    * **Rust/TS連携とプロトタイプ編集:** `String.__proto__.foo = foo;`のような動的編集がTS側から行われる場合、`foo`がSnowFallオブジェクトとして表現される必要があります。
        * **データフロー:** TS/JS側のオブジェクトは、Wasm境界を越えてRust側の `SnowObject`(またはその参照)として渡される必要があります。
        * **Wasm Interface:** Wasmインターフェースは、プロトタイプオブジェクトを操作するための専用関数(例: `update_prototype(type_id, property_name, new_object)`)を提供する必要があります。
* **Object型:** `Object`型はすべての型が継承する**最上位の基本型**(ルートクラス)です。すべてのSnowFallのインスタンスはこの`Object`型として扱えます(例: `String`は内部的に `Char[]` のように表現されます)。

### 3.2. 演算子と制御構文

#### 3.2.1. 等号・比較演算子

| 演算子 | 意味             | 補足                                       |
| :----- | :--------------- | :----------------------------------------- |
| `=`    | 代入             | すべての場合で代入に使用されます。         |
| `==`   | 型を無視した比較 | **暗黙の型変換**を伴う比較です。           |
| `===`  | 型比較も含む比較 | 型と値の両方を比較します(TypeScript方式)。 |

* **暗黙の型変換の優先順位:** `==`での型変換は、優先順位が高い型に合わせます。
    * **優先度順:** `String` > `Float` > `Int` > `Char` > `Boolean`
    * たとえば、`Int == Float` の場合、`Int`が`Float`に変換されてから比較されます。

#### 3.2.2. 論理演算子

| 演算子        | 意味             | 評価方法                                                                                      |
| :------------ | :--------------- | :-------------------------------------------------------------------------------------------- |
| `&&` / `\|\|` | 論理AND / 論理OR | **短絡評価**で計算し、結果は最後に評価されたオペランドの値を返します(JS/Python方式)。         |
| `and` / `or`  | 論理AND / 論理OR | **両辺を常に評価**し、結果を**必ず**`Boolean`で返します(副作用を持つコードの実行保証のため)。 |

#### 3.2.3. 特殊な計算

* **文字列の繰り返し:** 文字列と整数の乗算 (`*`) で文字列を繰り返します。
    * 例: `String stringB = stringA * 3;` は、`stringA`を3回繰り返した文字列を返却します(Pythonと同様の動作)。
* **文字列連結:** `+` 演算子を使用して文字列連結を行います。
    1. 連結の優先順位
        * **オペランドに一つでも`String`型が含まれる場合、`+`演算子はすべて文字列連結として動作します。**
        | 演算              | 挙動                                          | 例                 |
        | :---------------- | :-------------------------------------------- | :----------------- |
        | `String + Any`    | `Any`オペランドを`String`に変換し、連結する。 | `"A" + 1 → "A1"`   |
        | `Any + String`    | `Any`オペランドを`String`に変換し、連結する。 | `1 + "A" → "1A"`   |
        | `String + String` | 通常の文字列連結。                            | `"A" + "B" → "AB"` |
    2. 非文字列型との演算
        * オペランドに`String`が含まれない場合は、通常の算術演算として動作します(例: `Int + Float`は`Float`を返す)。
    3. `String`への型変換(暗黙的)
        * 文字列連結のために非文字列型(`Int`, `Float`, `Boolean`など)が`String`に変換される際、そのオブジェクトの`__to_string()`**特殊メソッド**(3.3. 節)が使用されます。
            * **プリミティブ型:** 標準の文字列表現に変換されます(例: `Int 10` $\rightarrow$ `"10"`, `Boolean True` $\rightarrow$ `"true"`)。
        * **カスタムオブジェクト:** `__to_string()` メソッドが実装されている必要があります。実装されていない場合、コンパイルエラー(または実行時エラー)とします。

#### 3.2.4. 制御構文

* **If文:** TypeScript方式で比較します。
    * 例: `if (foo === bar) { /* ... */ }`
* **For文:** 標準ループとforeachに対応します。
    * 標準ループ: `for (Int i=0;i<10;i++) { /* ... */ }`
    * ForEachループ:
        * **キー/インデックス取得 (`in`):** `for (Type keyOrIndex in Collection) { ... }` の形式。`Dictionary`では**キー**、`Array`では**インデックス**を順次取得します。
        * **値の取得 (`of`):** `for (Type value of Collection) { ... }` の形式。`Dictionary`および`Array`の**値**を順次取得します。
        * **キーと値の同時取得:** `for (Type1 key, Type2 value of Collection) { ... }` の形式。`Dictionary`および`Array`(インデックスと値)で使用可能とします。

### 3.3. クラスとオーバーライド

* **コンストラクタ:** クラスのコンストラクタは `__init__()` です。
* **演算子オーバーロード:** `__add__()`、`__sub__()`、`__mul__()`、`__div__()`、`__eq__()`、`__lt__()` などの特殊メソッドを定義し、演算をオーバーロード可能です。**厳密比較 (`===`) はオーバーロード不可**とし、組み込みロジックのみに依存します。

* **オーバーライドとラッパー:**
    * オーバーライドするメソッドの前には `@override` を記載します。
    * その他のラッパー(デコレーター)も `@foo` 形式で定義します。
* **デコレーター (`@foo`) の仕様とコンパイル時の処理:** デコレーターは、クラス、メソッド、プロパティにメタプログラミングの機能を提供し、静的検証や実行時の挙動をカスタマイズするための機構です。
    1. デコレーターの構文と適用対象
        * **構文:** `@<デコレーター名>(<引数リスト>)`
        * **適用対象:** `Class`定義、`Function` / `Sub`定義、クラス内のメソッド、クラス内のプロパティ/フィールド。
    2. デコレーターの実体(Rust / TS 連携)
        * デコレーターの実体は、Rust側でコンパイル時に実行される関数として定義します。
            1. **コンパイル時の処理:** ソースコードのAST上でデコレーターが検出された場合、Rustコンパイラはデコレーター関数を呼び出します。
            2. **入力 (引数):** デコレーター関数には、デコレートされる要素の情報が渡されます。
                * **クラスデコレーター:** デコレートされるクラスの定義オブジェクト。
                * **ソッドデコレーター:** メソッドの定義オブジェクト、メソッドが属するクラスの定義オブジェクト。
            3. **出力 (返り値):** デコレーター関数は、**オリジナルの要素をラップまたは修正した新しい定義オブジェクト**を返却します。
                * **ロジックの注入:** たとえば、`@logging` デコレーターは、元のメソッドの実行前後にログ出力処理を追加した新しいメソッド定義を生成し、返します。
            4. **実行時のオーバーライド:** デコレーターによってラップされた新しいメソッド定義が、最終的な中間コードに書き込まれます。
                * **設計上の決定 (`@override`):** `@override`は、他のデコレーターとは異なり、コンパイル時に親クラスのシグネチャとの一致を**検証するためだけ**に使用され、実行時のコード生成には影響を与えない特殊なデコレーター(**マーカー**)として扱われます。
        * デコレーターの型チェック
            * デコレーターによって新しい定義が生成された場合、コンパイラは生成された**新しい定義**に対して後続の静的検証を行います。
                * 例: `@memoize`デコレーターがメソッドの戻り値の型を変更しないことを、コンパイラは検証する必要があります。


### 3.4. 型変換と数値リテラル

* **型変換(キャスト):** C言語形式のキャストが可能です。
    * 例: `(int)"0x13"`
* **C言語形式キャスト (型名)値 の仕様:**
  * `Int x = (Int)"123";` のようなC言語形式のキャストは、明示的な型変換として扱います。
    |    変換元 (値)    | 変換先 (型名) | 成功条件と挙動                                                                                                         |
    | :---------------: | :-----------: | :--------------------------------------------------------------------------------------------------------------------- |
    | String (数値形式) |   Int,Float   | 形式が有効な数値表現であること。変換に失敗した場合、RuntimeErrorを発生させる。                                         |
    |    Int, Float     |    String     | 数値を文字列形式に変換する。                                                                                           |
    |    Int, Float     |    Boolean    | 0または0.0はFalse、それ以外はTrueに変換する。                                                                          |
    |        Int        |     Char      | Unicodeコードポイントとして解釈し、対応する文字に変換する。コードポイントが範囲外の場合、RuntimeError。                |
    |      Object       |      Any      | __to_type() のような特殊メソッドが定義されている場合、そのメソッドのロジックに従う。定義がない場合はコンパイルエラー。 |
* **型変換(メソッド):** 標準ライブラリのメソッドも利用可能です。
    * 例: `Integer.parse("0b101")`
* **メソッド形式キャスト Integer.parse() の仕様:**
    * `Integer.parse("0b101")` のようなメソッド形式のキャストは、より複雑な変換(基数指定など)に特化します。
        * **形式:** `[型名].parse(value, [base])`
        * **特徴:** parse メソッドは、C言語形式のキャストとは異なり、**基数を指定できる**などのオプション引数を許容します。また、標準ライブラリの型が提供する**静的メソッド**として実装されます。
        * **エラー処理:** 変換不能な場合、標準ライブラリメソッド内でエラーを捕捉し、SnowFallError を出力します。
* **数値リテラル:** 以下の基数で指定可能です。また、浮動小数点数の特殊な形式にも対応します。
    * 10進数: `99`
    * 16進数: `0xff`
    * 2進数: `0b11`
    * **アンダースコア`_`の解釈:** `10_000` のようなアンダースコアは、可読性向上のためのシンタックスシュガーであり、字句解析の段階で完全に無視されます。
        * **規則:**
            * アンダースコアは、数値リテラルの**数字と数字の間**にのみ挿入可能です。
            * 数値リテラルの先頭、末尾、または小数点(`.`)の直前/直後に配置された場合、`コンパイルエラー`とします。
            * 例: `100_000` は許容。 `_100` や `1.0_` はエラー。


### 3.5. 再帰制御

* **関数とサブルーチン:**
    * **関数 (`function`):** 戻り値の型を必須とします。
        | 項目 | 構文                                                    | 説明                                                                     |
        | :--- | :------------------------------------------------------ | :----------------------------------------------------------------------- |
        | 定義 | `function <戻り値の型> <関数名> (<引数リスト>) { ... }` | 戻り値の型は、プリミティブ型、オブジェクト型、または`Object`のいずれか。 |
        | 例   | `function Int sum(Int a, Int b) { return a + b; }`      | 戻り値の型を省略することはできません。                                   |
        | 匿名 | `function Int (Int a) { return a * 2; }`                | 匿名関数(クロージャ)もサポートします。                                   |
    * **サブルーチン (`sub`):** 戻り値を持たないため、戻り値の型指定は禁止されます。
        | 項目 | 構文                                          | 説明                                                 |
        | :--- | :-------------------------------------------- | :--------------------------------------------------- |
        | 定義 | `sub <サブルーチン名> (<引数リスト>) { ... }` | 戻り値を持つ`return`文は使用できません。             |
        | 例   | `sub log(String message) { print(message); }` | 戻り値は常に`Void`(または`Unit`)型として扱われます。 |
* **再帰:** `function` および `sub` は再帰可能です。
* **再帰上限:** 再帰回数の上限は、`SnowFall`実行コンストラクタオプションで個別に指定可能です。
    * 初期値は **10,000回** です。
    * 上限チェックは **Rust側** で行います。

### 3.6. Rust-TS 双方向連携プロトコル(実行時連携)
実行ステップ 6(Rust実行) と 7(TS連携) 間で発生する、非同期的な命令の受け渡しと結果の返却の仕様を定義します。

1. **基本設計:** 同期的な要求と非同期的な実行
    * Rust (Wasm) はシングルスレッドで動作するため、TS側の関数を呼び出す際は、Rust側の実行を一時停止し、TS側からの結果が返却されるのを待つ同期的なブロック呼び出しとして扱います。
2. **TS側が提供するインターフェース(外部環境):** TS側は、Rustが中間コード実行中に呼び出すためのグローバルなコールバック関数(またはクラスメソッド)を提供します。
    * **インターフェース名:** `SnowFallRuntime.invokeHostFunction(request: HostRequest): HostResponse`
    * **処理:** Rust (Wasm) のメモリ空間から、要求データ (`HostRequest`) のポインタとサイズをTSに渡し、TSが実行結果 (`HostResponse`) のデータをポインタでRustに返す。

    1. **データ構造1:** ホスト要求 (`HostRequest`)
        * RustからTSへ渡される要求データ(Wasmメモリ上に配置され、ポインタで参照される)の構造を定義します。これは主にJSONまたはバイナリ形式で表現されます。
        |    プロパティ     |        型        | 説明                                                              | 例                                              |
        | :---------------: | :--------------: | :---------------------------------------------------------------- | :---------------------------------------------- |
        |    `operation`    |      String      | TS側で実行すべき操作の種類。                                      | `"console.log", "dom.query", "ts_library.call"` |
        |      `args`       | Array<SnowValue> | SnowFallの型に変換された引数のリスト。                            | `["Hello, World", 123]`                         |
        | `requires_return` |     Boolean      | TS側からの結果を待つ必要があるか(Subからの呼び出しの場合は不要)。 | `true`                                          |
    2. **データ構造2:** ホスト応答 (`HostResponse`)
        * TS側が処理を終えてRustへ返却する応答データ(Wasmメモリ上に配置される)の構造を定義します。
        |  プロパティ  |      型       | 説明                                                     |
        | :----------: | :-----------: | :------------------------------------------------------- |
        |   `status`   |    String     | 実行結果の状態。`"OK"`、`"ERROR"`。                      |
        |   `result`   |   SnowValue   | 実行が成功した場合の戻り値(SnowFall型に変換されたもの)。 |
        | `error_info` | SnowFallError | 実行中にTS側でエラーが発生した場合のエラー情報。         |
3. **`SnowFall`言語側での呼び出し:** SnowFall言語のライブラリ(たとえば、標準ライブラリの`print`関数)は、内部的にこのプロトコルをラップします。
    * **SnowFallコード:** `print("Hello");`
    * **中間コード:** `CALL_HOST("console.log", ["Hello"])`
    * **Rust実行:** ホスト関数を呼び出し、`HostRequest { operation: "console.log", ... }` をTSに送信。
    * **TS実行:** `console.log("Hello")`を実行し、`HostResponse { status: "OK", result: Void }`をRustに返却。
4. **外部ライブラリ連携:** ユーザーがTypeScriptで作成した外部ライブラリをSnowFallから利用する場合、そのライブラリはTS側の`SnowFallRuntime`に登録される必要があります。
    * **登録:** TS側で、Rustからの呼び出しに対応するラッパー関数を定義し、`SnowFallRuntime`に登録します。
    * **型変換の責務:** 引数/戻り値の**SnowFall型** $\rightleftharpoons$ **TypeScript型**への変換は、TS側のラッパー関数(または`SnowFallRuntime`)が担います。
